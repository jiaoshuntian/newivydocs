## Contributing

IvorySQL is maintained by a core team of developers with commit rights to the main IvorySQL repository on GitHub. At the same time, the project eagerly welcomes contributions from anybody in the wider IvorySQL community.

### Getting started

IvorySQL is developed on GitHub. Anyone wishing to contribute will need a GitHub account and familiarity with Git tools and workflow. It is also recommended that you follow the developer's mailing list, since some contributions may generate more detailed discussions there.

Once you have a GitHub account, fork the repository so that you have your own private copy to start hacking on and to use as the source for pull requests.

### Licensing of IvorySQL contributions

**If the contribution you are submitting is original work**, you can assume that IvorySQL will release it as part of an overall IvorySQL release available to downstream consumers under the Apache License, Version 2.0.

**If the contribution is NOT original work**, you must indicate the name of the license and make sure it is similar in terms to the Apache License 2.0. The Apache Software Foundation maintains a list of compatible licenses under Category A. In addition, you may be required to provide proper attributions.

<Callout type="warn">
It is never a good idea to remove licensing headers from work that is not your own. Even if you are using only part of a file that originally had a licensing header, err on the side of preserving it.
</Callout>

If you are unsure about the licensing implications of your contribution, reach out on the developer mailing list before submitting.

## Coding guidelines

Your chances of getting feedback and seeing your code merged greatly depend on how granular your changes are. If you have a larger change in mind, engage on the developer's mailing list first and share your proposal before spending significant time writing code. Even when your proposal is validated by the community, doing the actual work as a series of small, self-contained commits makes the reviewer's job easier and increases the timeliness of feedback.

### Style conventions

When it comes to C and C++ parts of IvorySQL, the project follows PostgreSQL Coding Conventions. In addition:

- For **C** and **Perl** code, run `pgindent` if necessary.
- Use `git diff --color` when reviewing your changes to catch spurious whitespace issues.

### Test coverage

All new functionality contributed to IvorySQL must be covered by regression tests. If you are uncertain about how to test or document your work, raise the question on the `ivorysql-hackers` mailing list.

At a minimum, always run the following before submitting:

```bash
make installcheck-world
```

This ensures you are not breaking anything.

### Changes applicable to upstream PostgreSQL

If the change you are working on touches functionality that is common between PostgreSQL and IvorySQL, you may be asked to forward-port it to PostgreSQL. This keeps the delta between the two projects small and allows changes relevant to PostgreSQL to benefit from a broader upstream review.

## Patch submission

Once you are ready to share your work with the IvorySQL core team and community, push all commits to a branch in your forked repository and send a pull request to the official IvorySQL repository.

---

## Patch review

A submitted pull request with passing validation checks is assumed to be available for peer review. Peer review ensures contributions are of high quality and align with the roadmap and community expectations. Every member of the IvorySQL community is encouraged to review pull requests and provide feedback — you do not have to be a core team member to do so.

If the peer review reaches a consensus that you need to modify your pull request, push additional commits to the same branch. Those commits will be visible to all reviewers.

**A peer review converges when** it receives at least one `+1` and no `-1` votes from participants. At that point, expect a core team member to merge your changes.

If you are not getting expected feedback after a few days, add a comment on the pull request or send an email to the mailing list.

---

## Commit access

### Direct commits to the repository

Core team members may occasionally commit directly to the repository without going through the pull request workflow. This is reserved for small changes only. The rule of thumb: if the change touches any functionality that may result in a test failure, it must go through a pull request. If the change is in the non-functional part of the codebase (such as fixing a typo in a comment), core team members may commit directly.
