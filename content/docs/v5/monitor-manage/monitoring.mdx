# Monitor Your Database

IvorySQL provides a rich set of monitoring capabilities, ranging from basic Unix process inspection to detailed statistics views, wait event analysis, progress reporting for long-running commands, and dynamic tracing via DTrace/SystemTap. This guide covers all available monitoring mechanisms.

## Monitoring Database Activity

### Standard Unix Tools

On most Unix platforms, IvorySQL modifies its command title as reported by `ps`, so individual server processes can be identified:

```bash
$ ps auxww | grep ^postgres
postgres  15551  0.0  0.1  57536  7132 pts/0    S    18:02   0:00 postgres -i
postgres  15554  0.0  0.0  57536  1184 ?        Ss   18:02   0:00 postgres: background writer
postgres  15555  0.0  0.0  57536   916 ?        Ss   18:02   0:00 postgres: checkpointer
postgres  15556  0.0  0.0  57536   916 ?        Ss   18:02   0:00 postgres: walwriter
postgres  15557  0.0  0.0  58504  2244 ?        Ss   18:02   0:00 postgres: autovacuum launcher
postgres  15558  0.0  0.0  17512  1068 ?        Ss   18:02   0:00 postgres: stats collector
postgres  15582  0.0  0.0  58772  3080 ?        Ss   18:04   0:00 postgres: joe runbug 127.0.0.1 idle
postgres  15606  0.0  0.0  58772  3052 ?        Ss   18:07   0:00 postgres: tgl regression [local] SELECT waiting
postgres  15610  0.0  0.0  58772  3056 ?        Ss   18:07   0:00 postgres: tgl regression [local] idle in transaction
```

The first process is the primary server process. The next four are background worker processes. Each remaining process handles one client connection, displaying:

```
postgres: user database host activity
```

The activity indicator can be `idle` (waiting for a client command), `idle in transaction` (waiting inside a `BEGIN` block), or a command type name such as `SELECT`. `waiting` is appended if the server process is waiting on a lock held by another session.

If `cluster_name` has been configured, it also appears in `ps` output:

```bash
$ psql -c 'SHOW cluster_name'
 cluster_name
--------------
 server1
(1 row)

$ ps aux|grep server1
postgres   27093  0.0  0.0  30096  2752 ?        Ss   11:34   0:00 postgres: server1: background writer
```

If you have turned off `update_process_title`, the activity indicator is not updated after process launch.

<Callout type="tip">
Solaris requires special handling: use `/usr/ucb/ps` with two `w` flags, and ensure the original `postgres` command line is shorter than each server process's `ps` display.
</Callout>

### The Cumulative Statistics System

IvorySQL's cumulative statistics system collects and reports information about server activity, including table and index access counts, row counts, vacuum and analyze actions, and optionally function call tracking and timing.

#### Statistics Collection Configuration

Since collecting statistics adds overhead, you can enable or disable specific categories via configuration parameters in `postgresql.conf`:

| Parameter | Purpose |
|-----------|---------|
| `track_activities` | Monitor the current command being executed by any server process |
| `track_counts` | Collect cumulative statistics about table and index accesses |
| `track_functions` | Track usage of user-defined functions |
| `track_io_timing` | Monitor block read and write times |
| `track_wal_io_timing` | Monitor WAL write times |

These can also be toggled per session using `SET` (restricted to superusers).

Cumulative statistics are collected in shared memory. When a server shuts down cleanly, a permanent copy is stored in the `pg_stat` subdirectory, allowing statistics to persist across restarts. After an unclean shutdown, all statistics counters are reset.

#### Viewing Statistics

Several predefined views show the current and accumulated state of the system. The information does not update instantaneously -- each server process flushes accumulated statistics to shared memory just before going idle (at least once per second).

<Callout type="info">
By default, statistics are cached until the end of the current transaction. This allows you to perform several queries and correlate results without the numbers changing underneath you. Set `stats_fetch_consistency` to `snapshot` to minimize skew, or `none` if you only access statistics once.
</Callout>

Some dynamic statistics views are security-restricted. Ordinary users can only see information about their own sessions. Superusers and roles with `pg_read_all_stats` privileges can see all information.

#### Dynamic Statistics Views

| View | Description |
|------|-------------|
| `pg_stat_activity` | One row per server process, showing state and current query |
| `pg_stat_replication` | One row per WAL sender process, replication statistics |
| `pg_stat_wal_receiver` | WAL receiver statistics from connected server |
| `pg_stat_recovery_prefetch` | Statistics about blocks prefetched during recovery |
| `pg_stat_subscription` | Information about subscription workers |
| `pg_stat_ssl` | SSL information per connection |
| `pg_stat_gssapi` | GSSAPI authentication and encryption per connection |
| `pg_stat_progress_analyze` | Current progress of `ANALYZE` operations |
| `pg_stat_progress_create_index` | Current progress of `CREATE INDEX` or `REINDEX` |
| `pg_stat_progress_vacuum` | Current progress of `VACUUM` operations |
| `pg_stat_progress_cluster` | Current progress of `CLUSTER` or `VACUUM FULL` |
| `pg_stat_progress_basebackup` | Progress of streaming base backups |
| `pg_stat_progress_copy` | Progress of `COPY` operations |

### pg_stat_activity

The `pg_stat_activity` view has one row per server process:

| Column | Type | Description |
|--------|------|-------------|
| `datid` | `oid` | OID of the connected database |
| `datname` | `name` | Name of the connected database |
| `pid` | `integer` | Process ID of this backend |
| `leader_pid` | `integer` | Process ID of the parallel group leader (if applicable) |
| `usesysid` | `oid` | OID of the logged-in user |
| `usename` | `name` | Name of the logged-in user |
| `application_name` | `text` | Name of the connected application |
| `client_addr` | `inet` | IP address of the client (null for Unix socket or internal process) |
| `client_hostname` | `text` | Host name from reverse DNS lookup (if `log_hostname` is enabled) |
| `client_port` | `integer` | TCP port number, or `-1` for Unix socket |
| `backend_start` | `timestamp with time zone` | When this process started |
| `xact_start` | `timestamp with time zone` | When current transaction started |
| `query_start` | `timestamp with time zone` | When the currently active query started |
| `state_change` | `timestamp with time zone` | When the `state` was last changed |
| `wait_event_type` | `text` | Type of event the backend is waiting for (if any) |
| `wait_event` | `text` | Wait event name if currently waiting |
| `state` | `text` | Overall state: `active`, `idle`, `idle in transaction`, `idle in transaction (aborted)`, `fastpath function call`, `disabled` |
| `backend_xid` | `xid` | Top-level transaction identifier |
| `backend_xmin` | `xid` | The backend's `xmin` horizon |
| `query_id` | `bigint` | Identifier of the most recent query |
| `query` | `text` | Text of the most recent query (truncated at 1024 bytes by default) |
| `backend_type` | `text` | Type: `autovacuum launcher`, `autovacuum worker`, `client backend`, `background writer`, etc. |

<Callout type="info">
The `wait_event` and `state` columns are independent. A backend in `active` state may or may not be waiting on an event. If `state` is `active` and `wait_event` is non-null, a query is executing but is being blocked somewhere.
</Callout>

#### Wait Event Types

| Type | Description |
|------|-------------|
| `Activity` | Server process is idle in its main loop |
| `BufferPin` | Waiting for exclusive access to a data buffer |
| `Client` | Waiting for activity on a client socket |
| `Extension` | Waiting for a condition defined by an extension |
| `IO` | Waiting for an I/O operation to complete |
| `IPC` | Waiting for interaction with another server process |
| `Lock` | Waiting for a heavyweight lock (tables, relations, etc.) |
| `LWLock` | Waiting for a lightweight lock (shared memory data structures) |
| `Timeout` | Waiting for a timeout to expire |

Example -- view wait events:

```sql
SELECT pid, wait_event_type, wait_event FROM pg_stat_activity WHERE wait_event IS NOT NULL;
 pid  | wait_event_type | wait_event
------+-----------------+------------
 2540 | Lock            | relation
 6644 | LWLock          | ProcArray
(2 rows)
```

### pg_stat_replication

Shows statistics about replication to each connected standby server (only directly connected standbys):

| Column | Type | Description |
|--------|------|-------------|
| `pid` | `integer` | Process ID of WAL sender |
| `usesysid` | `oid` | OID of the user |
| `usename` | `name` | Name of the user |
| `application_name` | `text` | Name of the connected application |
| `client_addr` | `inet` | IP address of the standby |
| `client_port` | `integer` | TCP port number |
| `backend_start` | `timestamp with time zone` | When the client connected |
| `backend_xmin` | `xid` | Standby's `xmin` horizon from hot_standby_feedback |
| `state` | `text` | `startup`, `catchup`, `streaming`, `backup`, `stopping` |
| `sent_lsn` | `pg_lsn` | Last WAL location sent |
| `write_lsn` | `pg_lsn` | Last WAL location written to disk by standby |
| `flush_lsn` | `pg_lsn` | Last WAL location flushed to disk by standby |
| `replay_lsn` | `pg_lsn` | Last WAL location replayed on standby |
| `write_lag` | `interval` | Delay for `remote_write` synchronous commit |
| `flush_lag` | `interval` | Delay for `on` synchronous commit |
| `replay_lag` | `interval` | Delay for `remote_apply` synchronous commit |
| `sync_priority` | `integer` | Priority for synchronous standby selection |
| `sync_state` | `text` | `async`, `potential`, `sync`, `quorum` |
| `reply_time` | `timestamp with time zone` | Send time of last reply from standby |

<Callout type="info">
Lag times are not predictions of catch-up time. When the standby has caught up completely, lag columns show the time taken for the most recent reported WAL, then revert to NULL after a short idle period.
</Callout>

### View Locks

The `pg_locks` system table shows information about outstanding locks in the lock manager. It can be used to:

- View all currently outstanding locks, locks on specific relations, or locks held by a specific session
- Determine which relation has the most ungranted locks (a potential contention source)
- Assess the impact of lock contention on database performance

## Dynamic Tracing

### Using Probes

Example DTrace script for analyzing transaction counts:

```c
#!/usr/sbin/dtrace -qs

postgresql$1:::transaction-start
{
      @start["Start"] = count();
      self->ts  = timestamp;
}

postgresql$1:::transaction-abort
{
      @abort["Abort"] = count();
}

postgresql$1:::transaction-commit
/self->ts/
{
      @commit["Commit"] = count();
      @time["Total time (ns)"] = sum(timestamp - self->ts);
      self->ts=0;
}
```

Running the script:

```bash
# ./txn_count.d `pgrep -n postgres` or ./txn_count.d <PID>
^C

Start                                          71
Commit                                         70
Total time (ns)                        2312105013
```

<Callout type="info">
SystemTap uses double underscores in place of hyphens for probe names (e.g., `transaction__start` instead of `transaction-start`). This is expected to be fixed in future SystemTap releases.
</Callout>

### Defining New Probes

To add a custom probe:

1. Decide on probe names and data to expose
2. Add probe definitions to `src/backend/utils/probes.d`
3. Include `pg_trace.h` and insert `TRACE_POSTGRESQL` probe macros
4. Recompile and verify

**Example: trace all new transactions by transaction ID:**

1. Name the probe `transaction-start` with a `LocalTransactionId` parameter
2. Add to `probes.d`:

```
probe transaction__start(LocalTransactionId);
```

3. At the desired location in source code:

```c
TRACE_POSTGRESQL_TRANSACTION_START(vxid.localTransactionId);
```

4. Verify after recompiling:

```text
# dtrace -ln transaction-start
   ID    PROVIDER          MODULE           FUNCTION NAME
18705 postgresql49878     postgres     StartTransactionCommand transaction-start
```

<Callout type="warn">
When `--enable-dtrace` is used, probe macro arguments are evaluated whenever control passes through the macro, even if no tracing is active. Protect expensive function calls:

```c
if (TRACE_POSTGRESQL_TRANSACTION_START_ENABLED())
    TRACE_POSTGRESQL_TRANSACTION_START(some_function(...));
```
</Callout>

## Monitoring Disk Usage

### Determining Disk Usage

Each table has a primary heap disk file. Tables with wide-value columns may also have a TOAST file. Each table and index is stored in a separate disk file (possibly multiple files if exceeding 1 GB).

Monitor disk space using SQL functions (recommended), the `oid2name` module, or system catalog inspection:

```sql
SELECT pg_relation_filepath(oid), relpages FROM pg_class WHERE relname = 'customer';

 pg_relation_filepath | relpages
----------------------+----------
 base/16384/16806     |       60
(1 row)
```

Each page is typically 8 kilobytes. (`relpages` is only updated by `VACUUM`, `ANALYZE`, and certain DDL commands.)

Show TOAST table usage:

```sql
SELECT relname, relpages
FROM pg_class,
     (SELECT reltoastrelid FROM pg_class WHERE relname = 'customer') AS ss
WHERE oid = ss.reltoastrelid OR
      oid = (SELECT indexrelid FROM pg_index WHERE indrelid = ss.reltoastrelid)
ORDER BY relname;

       relname        | relpages
----------------------+----------
 pg_toast_16806       |        0
 pg_toast_16806_index |        1
```

Show index sizes:

```sql
SELECT c2.relname, c2.relpages
FROM pg_class c, pg_class c2, pg_index i
WHERE c.relname = 'customer' AND
      c.oid = i.indrelid AND
      c2.oid = i.indexrelid
ORDER BY c2.relname;

      relname      | relpages
-------------------+----------
 customer_id_index |       26
```

Find largest tables:

```sql
SELECT relname, relpages
FROM pg_class
ORDER BY relpages DESC;

       relname        | relpages
----------------------+----------
 bigtable             |     3290
 customer             |     3144
```

### Disk Full Failure

The most critical disk monitoring task is ensuring the disk does not become full. A filled WAL disk will cause the server to panic and shut down. A filled data disk won't corrupt data but may prevent useful activity.

If you cannot free space, consider moving database files to other file systems using tablespaces.

<Callout type="tip">
Some file systems perform badly when nearly full. Do not wait until the disk is completely full to take action.
</Callout>

If your system has per-user disk quotas, the database is subject to the quota of the user the server runs as. Exceeding the quota has the same effects as running out of disk space entirely.
