# PL/iSQL & Stored Procedures

PL/iSQL is IvorySQL's Oracle-compatible procedural language. It accepts the same syntax as Oracle's PL/SQL, so you can migrate stored functions and procedures from an Oracle database to IvorySQL with little or no modification.

<Callout type="tip">
**Migrating from Oracle?** Most PL/SQL function and procedure bodies work as-is. The main things to check are: `AUTHID` defaults, unsupported `SHARING` clauses (accepted but ignored), and Oracle-specific packages (`DBMS_*`) which need the IvorySQL equivalents.
</Callout>

## PL/iSQL vs PL/pgSQL

| | PL/iSQL | PL/pgSQL |
|--|---------|---------|
| **Use when** | Migrating Oracle code, targeting Oracle syntax | Writing new PostgreSQL-native logic |
| **Variable declaration** | Between `AS`/`IS` and `BEGIN`, no `DECLARE` keyword | Requires `DECLARE` block |
| **Return type keyword** | `RETURN integer` | `RETURNS integer` |
| **Statement terminator in psql** | `/` | `;` |
| **Null-parameter call** | No parentheses needed | Requires `()` |

Both languages coexist in IvorySQL. Choose PL/iSQL when you need Oracle syntax; choose PL/pgSQL for new code.

## Write your first function

```sql
CREATE OR REPLACE FUNCTION greet(name IN VARCHAR2)
RETURN VARCHAR2 IS
BEGIN
  RETURN 'Hello, ' || name || '!';
END;
/

SELECT greet('IvorySQL') FROM dual;
--       greet
-- ----------------
--  Hello, IvorySQL!
```

Key differences from PL/pgSQL to notice:
- `RETURN varchar2` (not `RETURNS`)
- Variable declarations go directly after `IS`, no `DECLARE`
- End with `/` in psql (not `;`)

## Write your first procedure

```sql
CREATE OR REPLACE PROCEDURE log_message(msg IN VARCHAR2)
IS
BEGIN
  RAISE NOTICE '%', msg;
END;
/

CALL log_message('Migration complete');
-- NOTICE:  Migration complete
```

## Migrate an Oracle function

Here is a typical Oracle function and its IvorySQL equivalent:

**Oracle original:**
```sql
CREATE OR REPLACE FUNCTION calc_bonus(
  salary IN NUMBER,
  rate   IN NUMBER
) RETURN NUMBER AS
  bonus NUMBER;
BEGIN
  bonus := salary * rate;
  RETURN bonus;
END;
/
```

**IvorySQL — identical syntax, works without changes:**
```sql
CREATE OR REPLACE FUNCTION calc_bonus(
  salary IN NUMBER,
  rate   IN NUMBER
) RETURN NUMBER AS
  bonus NUMBER;
BEGIN
  bonus := salary * rate;
  RETURN bonus;
END;
/
```

## Common syntax patterns

### Functions with multiple parameters

```sql
CREATE OR REPLACE FUNCTION ora_func(num1 IN int, num2 IN int)
RETURN int
AS
  num3 int := 10;
  num4 int := 10;
  num5 int;
BEGIN
  num3 := num1 + num2;
  num4 := num1 * num2;
  num5 := num3 * num4;
  RETURN num5;
END;
/

SELECT ora_func(5, 9) FROM dual;
--  ora_func
-- ----------
--       630
```

### OUT parameters

```sql
CREATE OR REPLACE PROCEDURE get_user(
  user_id  IN  int,
  username OUT VARCHAR2
) IS
BEGIN
  SELECT name INTO username FROM users WHERE id = user_id;
END;
/
```

See [OUT Parameters](/docs/v5/oracle-compatibility/out-parameter) for full usage including `NOCOPY`.

### Call a function or procedure without parentheses

```sql
CREATE OR REPLACE FUNCTION get_timestamp RETURN TIMESTAMP IS
BEGIN
  RETURN SYSTIMESTAMP;
END;
/

-- No parentheses when there are no parameters
SELECT get_timestamp FROM dual;
```

### View Oracle-style catalog information

```sql
-- List all procedures and functions visible to the current user
SELECT object_name, object_type, status
FROM user_procedures;

-- View function source
SELECT text FROM user_source WHERE name = 'CALC_BONUS' ORDER BY line;
```

## Syntax reference

### Function declaration options

| Clause | Behavior in IvorySQL |
|--------|----------------------|
| `EDITIONABLE` / `NONEDITIONABLE` | Accepted, ignored (editions not supported) |
| `SHARING = METADATA \| NONE` | Accepted, ignored |
| `AUTHID CURRENT_USER` | Supported — function runs as caller |
| `AUTHID DEFINER` | Default — function runs as owner |
| `DETERMINISTIC` | Supported — maps to PostgreSQL `IMMUTABLE` |
| `PARALLEL_ENABLE` | Accepted, ignored |
| `RESULT_CACHE` | Accepted, ignored |
| `PIPELINED` | Accepted, ignored |
| `SQL_MACRO` | Accepted, ignored |
| `ACCESSIBLE BY (...)` | Accepted, ignored |
| `DEFAULT COLLATION` | Accepted, ignored |

### ALTER FUNCTION / ALTER PROCEDURE

```sql
ALTER FUNCTION test_func COMPILE;
ALTER FUNCTION test_func COMPILE DEBUG;
ALTER FUNCTION public.test_func NONEDITIONABLE;

ALTER PROCEDURE test_proc EDITIONABLE;
ALTER PROCEDURE test_proc COMPILE;
```

## Advanced clause syntax

### ACCESSIBLE BY — restrict callers

Control which program units may call a function or procedure. IvorySQL accepts the syntax but does not enforce the restriction — it is parsed for Oracle migration compatibility.

```sql
-- Allow access only by a specific function and procedure
CREATE OR REPLACE FUNCTION get_salary(emp_id IN int) RETURN NUMBER
SHARING = NONE AUTHID DEFINER
ACCESSIBLE BY ( FUNCTION hr.calc_bonus, PROCEDURE hr.update_payroll )
IS
  v_salary NUMBER;
BEGIN
  SELECT salary INTO v_salary FROM employees WHERE id = emp_id;
  RETURN v_salary;
END;
/

-- Multiple accessor types: FUNCTION, PROCEDURE, PACKAGE, TRIGGER, TYPE
CREATE OR REPLACE FUNCTION internal_lookup RETURN VARCHAR2
AUTHID DEFINER
ACCESSIBLE BY ( FUNCTION app.caller_fn, PROCEDURE app.caller_proc,
                PACKAGE app.utils, TRIGGER app.audit_trg, TYPE app.obj_type )
IS
BEGIN
  RETURN 'ok';
END;
/
```

### PARALLEL_ENABLE with partitioning

For functions used in parallel queries, you can specify how rows are partitioned among parallel execution servers:

```sql
CREATE OR REPLACE FUNCTION score_row(emp_id IN int) RETURN NUMBER
PARALLEL_ENABLE ( PARTITION emp_id BY RANGE ( dept_id, salary )
                  CLUSTER emp_id BY ( dept_id ) )
IS
BEGIN
  RETURN 1;
END;
/
```

<Callout type="tip">
`PARALLEL_ENABLE` with `PARTITION` syntax is accepted by IvorySQL's parser for Oracle compatibility. The partitioning hint itself has no effect — IvorySQL's query planner controls parallelism independently.
</Callout>

### RESULT_CACHE with RELIES_ON

```sql
-- Cache result, invalidate when specified tables change
CREATE OR REPLACE FUNCTION get_config(key IN VARCHAR2) RETURN VARCHAR2
RESULT_CACHE RELIES_ON ( app_config, app_defaults )
IS
  v_val VARCHAR2(256);
BEGIN
  SELECT value INTO v_val FROM app_config WHERE name = key;
  RETURN v_val;
END;
/

-- RESULT_CACHE RELIES_ON () with empty list is also accepted
CREATE OR REPLACE FUNCTION static_value RETURN int
RESULT_CACHE RELIES_ON ()
IS
BEGIN
  RETURN 42;
END;
/
```

### AGGREGATE USING

Associate a function with a custom aggregate implementation:

```sql
CREATE OR REPLACE FUNCTION agg_handler RETURN NUMBER
AGGREGATE USING my_schema.my_aggregate_type
IS
BEGIN
  RETURN 1;
END;
/
```

### Comments in function bodies

Both Oracle comment styles are supported inside PL/iSQL function and procedure bodies:

```sql
CREATE OR REPLACE FUNCTION commented_func(n IN int) RETURN int IS
  -- single-line comment
  result int; /* inline comment */
BEGIN
  /*
   * Multi-line block comment.
   * Both styles work identically.
   */
  result := n * 2;
  RETURN result;
END;
/
```

### ALTER … COMPILE variants

```sql
-- Recompile only
ALTER FUNCTION my_func COMPILE;

-- Compile with debug info
ALTER FUNCTION my_func COMPILE DEBUG;

-- Compile with debug and session-level settings
ALTER FUNCTION my_func COMPILE DEBUG sd = mv;

-- Compile and reuse current session settings
ALTER FUNCTION my_func COMPILE DEBUG REUSE SETTINGS;

-- Same variants apply to procedures
ALTER PROCEDURE my_proc COMPILE;
ALTER PROCEDURE my_proc COMPILE DEBUG REUSE SETTINGS;

-- Edition control
ALTER FUNCTION my_func EDITIONABLE;
ALTER FUNCTION public.my_func NONEDITIONABLE;
```

## System views

Oracle-compatible catalog views are available for introspecting procedures and functions:

| View | Scope |
|------|-------|
| `DBA_PROCEDURES` | All procedures/functions in the database |
| `ALL_PROCEDURES` | Procedures/functions accessible to current user |
| `USER_PROCEDURES` | Procedures/functions owned by current user |
| `DBA_SOURCE` / `ALL_SOURCE` / `USER_SOURCE` | Source code |
| `DBA_ARGUMENTS` / `ALL_ARGUMENTS` / `USER_ARGUMENTS` | Parameter definitions |

## Backup and export

When `pg_dump` exports PL/iSQL functions and procedures, it appends `/` at the end of each definition — ensuring the exported SQL can be re-run without modification.
